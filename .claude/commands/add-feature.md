# 新機能の追加 (完全自動実行モード)

**重要:** このワークフローは、ユーザーの介入なしに、開始から完了まで完全に自動で実行されるように設計されています。各ステップは完了後、ただちに次のステップへ移行してください。思考の途中でユーザーに確認を求めたり、作業を中断したりしないでください。

**引数:** 機能名 (例: `/add-feature 業種別ヒートマップ`)

対象プロジェクト: jpx500_wave_analysis (Streamlit + pandas + Parquet/CSV + FastAPI)

---

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:
   - 機能名: `[引数で与えられた機能名]`
   - 日付: `[現在の日付をYYYYMMDD形式で取得]`
   - ステアリングディレクトリパス: `.steering/[日付]-[機能名]/`
2. 上記ステアリングディレクトリを作成する。
3. 以下の3つの空ファイルを作成する:
   - `[ステアリングディレクトリパス]/requirements.md`
   - `[ステアリングディレクトリパス]/design.md`
   - `[ステアリングディレクトリパス]/tasklist.md`

## ステップ2: プロジェクト理解

1. `INTEGRATION_PLAN.md` を読み、naibu-ryuho-app との統合方針（API連携 + ユニバーステーブル）を把握する。
2. `README.md` を読み、機能・実行方法・データフローを把握する。
3. `CLAUDE.md` が存在すればそれも読む。

## ステップ3: 既存パターンの調査

1. Grep ツールで機能名に関連するキーワードを検索する:
   ```bash
   Grep('[機能に関連するキーワード]', 'modules/')
   Grep('[機能に関連するキーワード]', 'batch/')
   Grep('[機能に関連するキーワード]', 'app.py')
   Grep('[機能に関連するキーワード]', 'api_server.py')
   ```
2. 検索結果を分析し、既存の実装パターン、命名規則、関数分割の仕方を特定する。

3. レイヤー配置の判断:
   - **株価取得・キャッシュ**: `modules/data_fetcher.py` を拡張、出力先は `data/cache/*.parquet`
   - **波形分類・指標計算**: `modules/wave_classifier.py` または `modules/strategy_engine.py` を拡張
   - **チャート描画**: `modules/chart_builder.py` (Plotly)
   - **設定・閾値**: `config/settings.py` の定数追加 / `config/strategy.yaml` のパラメータ追加
   - **Streamlit UI**: `app.py` にセクション追加
   - **REST API エンドポイント**: `api_server.py` に `@app.get(...)` 追加（naibu連携用）
   - **日次バッチ**: `batch/update.py` にステップ追加 / `batch/` 配下に新規スクリプト
   - **データ出力 (CSV)**: `data/*.csv` に書き出し、`config/settings.py` でパス定数化

## ステップ4: 計画フェーズ (ステアリングファイルの自動生成)

1. `Skill('steering')` を**計画モード**で実行し、ステップ1で作成した3つのファイルの内容を生成する。
2. **このステップが正常に完了したら、決して停止せず、ただちにステップ5に進むこと。**

## ステップ5: 実装ループ (tasklist.md の完全消化)

**このステップは、`tasklist.md` の全タスクが `[x]` になるまで自動で繰り返されるループ処理です。**
**このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

**ループ開始:**

1. タスクリストの読み込み:
   - `[ステアリングディレクトリパス]/tasklist.md` を読み込む。

2. 進捗の確認:
   - 未完了タスク (`[ ]`) が存在するか確認。
   - 存在しない場合: ループ完了とみなし、ただちに**ステップ6**へ。
   - 存在する場合: 次の処理へ。

3. タスクの実行:
   - `tasklist.md` の**先頭にある未完了タスク**を1つ特定する。
   - そのタスクを完了させるために必要な実装作業を実行する。
   - `Skill('steering')` を**実装モード**で使用する。

4. タスクリストの更新:
   - タスク完了後、`Edit` で `tasklist.md` の該当タスクを `[ ]` → `[x]` に変更。

5. ループ継続:
   - **ステップ5の先頭に戻り、処理を繰り返す。**

---
### ※ 実装ループ内の例外処理ルール ※

- **ルールA: タスクが大きすぎる場合**
  - 現在のタスクをより小さなサブタスクに分割し、元のタスクを削除して新しいサブタスクを挿入する。ループ継続。

- **ルールB: 技術的理由でタスクが不要になった場合**
  - `[x] ~~タスク名~~ (理由: [具体的な技術的理由])` の形式で更新。ループ継続。

- **❌ 絶対禁止の行為:**
  - 未完了タスクを「後でやる」で意図的にスキップすること
  - 理由なく未完了タスクを放置してループを終了させること
  - ユーザーに判断を仰ぐこと

---

## ステップ6: 自動テストの実行

以下を**順番に**実行し、全てパスすることを確認する。

### 6-1. 静的解析 (任意・ツールがインストール済みの場合)
```bash
ruff check modules/ batch/ api_server.py app.py 2>/dev/null || echo "ruff 未インストール、スキップ"
ruff format --check modules/ batch/ api_server.py app.py 2>/dev/null || true
mypy modules/ batch/ api_server.py --ignore-missing-imports 2>/dev/null || echo "mypy 未インストール、スキップ"
```

### 6-2. ユニットテスト
```bash
pytest -q 2>/dev/null || echo "pytest 未設定、スキップ"
```

新機能のスモークテストが存在せず、API またはバッチ追加の場合は、**先に簡易テスト（test_*.py）を作成してから実行する**。

最低限カバーすべき内容:
- **API 追加時**: FastAPI の `TestClient` で `200 OK` 確認 + レスポンス構造の assert
- **バッチ追加時**: 主要関数のユニットテスト（少量データ・モック）
- **modules追加時**: 入出力 DataFrame の形を assert

### 6-3. 動作確認 (手動起動)

API追加なら uvicorn で起動して curl 確認:
```bash
python -m uvicorn api_server:app --host 127.0.0.1 --port 8001 &
sleep 2
curl -s http://127.0.0.1:8001/api/health
# 追加エンドポイントを curl で叩く
kill %1
```

Streamlit UI追加なら起動して画面確認（手動）:
```bash
python -m streamlit run app.py
```

### 6-4. エラー時の対処
いずれかでエラーが発生したら、原因を分析→修正→失敗ステップから再実行。全て通るまで繰り返す。

**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

## ステップ7: 振り返りとドキュメント整合（迷子防止）

### ステップ7-A: INTEGRATION_PLAN.md の更新（naibu連携に影響する場合）

今回の変更が naibu-ryuho-app との連携API・ユニバーステーブル等に影響する場合、**正本** (`naibu-ryuho-app/INTEGRATION_PLAN.md`) を `Edit` で更新し、本ファイル `INTEGRATION_PLAN.md` にもコピー反映する。

### ステップ7-B: README.md の整合確認

新機能で公開エンドポイント・コマンド・設定が増えた場合、`README.md` の該当セクションを更新。

### ステップ7-C: tasklist.md に振り返りを記載

`Skill('steering')` を**振り返りモード**で実行し、`tasklist.md` 末尾に申し送り事項を追記:
- 実装完了日（YYYY-MM-DD 形式）
- 計画と実績の差分
- 学んだこと
- 次回への改善提案

### ステップ7-D: `/status` で整合確認

完了報告の前に、以下を心の中でチェック:
- [ ] `.steering/<今回>/tasklist.md` が 100% `[x]` になっているか
- [ ] 未コミット変更が残っていないか（`git status`）
- [ ] INTEGRATION_PLAN.md と naibu側のコピーがズレていないか

ズレていたら 7-A に戻る。

## 完了条件

このワークフローは、以下の全条件を満たした時点で自動完了:
- ステップ5: `tasklist.md` の全タスクが完了状態
- ステップ6: lint/typecheck/test の利用可能なコマンドがエラーなく成功
- ステップ7-A: INTEGRATION_PLAN.md (両ファイル) の整合確保
- ステップ7-B: 必要に応じて README.md を更新
- ステップ7-C: `tasklist.md` に振り返り記載
- ステップ7-D: `/status` の整合確認通過

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
